## Overview

The BPRM Portal connects three teams — Stakeholders, Business Analysts, and IT — on a single platform. It replaces fragmented email chains and manual Word documents with a structured, AI-powered workflow:

```
Stakeholder submits problem
        ↓
BA discusses requirements via Stream Chat
        ↓
AI analyses chat → extracts key points → generates BRD
        ↓
BA reviews, enhances, and approves BRD
        ↓
IT receives BRD → generates FRD → generates Test Cases
        ↓
Stakeholder performs UAT → sign-off
```

## AI Features

### BRD Generation Pipeline

1. **Key Point Extraction** (`POST /api/stream/channels/:requestId/generate-key-points`)
   - Annotates each chat message with domain signals (financial, API, compliance, security, etc.)
   - Single GPT-4o-mini call classifies all messages into: `REQUIREMENT | CONCERN | ACTION | GENERAL`
   - Domain-override rules force financial/API/compliance messages into REQUIREMENT even if GPT disagrees
   - TF-IDF keyword extraction runs client-side (no API cost)
   - Returns: requirements, concerns, action items, keywords, integration signals, BRD readiness score

2. **BRD Generation** (`POST /api/stream/channels/:requestId/generate-brd`)
   - Synthesises structured BRD document from extracted key points
   - GPT-4o-mini generates professional BA-language text for each section
   - Domain-template libraries formalise raw chat into proper risk register and action item language
   - Sections: Executive Summary, Scope, Stakeholders, Functional Requirements, Non-Functional Requirements, Risk Register, Action Items, BRD Readiness

3. **BRD Enhancement** (`POST /api/stream/brd-documents/:brdId/enhance`)
   - Takes an existing BRD and re-runs GPT generation with enrichment prompts
   - Adds missing sections, expands thin requirements, improves language quality

4. **FRD Generation** (`POST /api/stream/brd-documents/:brdId/generate-frd`)
   - Expands approved BRD into full Functional Requirements Document
   - Sections: Overview, Functional Specifications (with acceptance criteria + business rules), System Behavior (workflows), Data Requirements, UI Requirements, Integration Requirements, NFRs, Traceability Matrix

5. **Test Case Generation** (`POST /api/stream/frd-documents/:frdId/generate-test-cases`)
   - Generates comprehensive test suites from FRD functional specifications
   - Test types: Critical, System, UAT, Regression
   - Each case includes: steps, expected outcome, priority, type classification

### PDF Export

Both BRD and FRD/TC exports use a professional `window.print()` design system:
- Dark gradient cover pages with metadata grids
- Dark-header tables, priority-bordered cards, two-column acceptance criteria layout
- Print-colour-adjust for accurate colour reproduction
- Auto table-of-contents generation

## Database Schema

```sql
-- Core users
users (
  id              SERIAL PRIMARY KEY,
  email           TEXT UNIQUE NOT NULL,
  password_hash   TEXT NOT NULL,
  name            TEXT,
  role            TEXT NOT NULL,          -- 'stakeholder' | 'ba' | 'it'
  is_it_manager   BOOLEAN DEFAULT false,
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Business requests submitted by stakeholders
requests (
  id              SERIAL PRIMARY KEY,
  req_number      TEXT UNIQUE,            -- e.g. REQ-001
  title           TEXT NOT NULL,
  description     TEXT,
  category        TEXT,
  priority        TEXT,
  status          TEXT DEFAULT 'Open',
  submitted_by    INTEGER REFERENCES users(id),
  assigned_to     INTEGER REFERENCES users(id),
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- File attachments on requests
request_attachments (
  id              SERIAL PRIMARY KEY,
  request_id      INTEGER REFERENCES requests(id),
  filename        TEXT,
  storage_path    TEXT,
  uploaded_by     INTEGER REFERENCES users(id),
  uploaded_at     TIMESTAMPTZ DEFAULT NOW()
)

-- BRD documents (AI-generated)
brd_documents (
  id              SERIAL PRIMARY KEY,
  doc_id          TEXT UNIQUE,            -- e.g. BRD-2024-001
  request_id      INTEGER REFERENCES requests(id),
  version         TEXT DEFAULT '1.0',
  status          TEXT DEFAULT 'Draft',   -- Draft | In Review | Approved | Final
  content         JSONB NOT NULL,
  generated_by    INTEGER REFERENCES users(id),
  generated_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Peer reviews on BRDs
brd_reviews (
  id              SERIAL PRIMARY KEY,
  brd_document_id INTEGER REFERENCES brd_documents(id),
  reviewer_id     INTEGER REFERENCES users(id),
  status          TEXT,                   -- 'approved' | 'rejected' | 'pending'
  comment         TEXT,
  reviewed_at     TIMESTAMPTZ DEFAULT NOW()
)

-- Tracks which BRDs have been routed to IT
brd_it_submissions (
  id              SERIAL PRIMARY KEY,
  brd_document_id INTEGER REFERENCES brd_documents(id),
  submitted_by    INTEGER REFERENCES users(id),
  it_manager_id   INTEGER REFERENCES users(id),
  submitted_at    TIMESTAMPTZ DEFAULT NOW()
)

-- FRD documents (generated by IT from approved BRDs)
frd_documents (
  id              SERIAL PRIMARY KEY,
  doc_id          TEXT UNIQUE,
  brd_document_id INTEGER REFERENCES brd_documents(id),
  request_id      INTEGER REFERENCES requests(id),
  version         TEXT DEFAULT '1.0',
  status          TEXT DEFAULT 'Draft',   -- Draft | In Review | Approved
  content         JSONB NOT NULL,
  generated_by    INTEGER REFERENCES users(id),
  generated_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Test case suites (generated from FRDs)
test_case_documents (
  id              SERIAL PRIMARY KEY,
  doc_id          TEXT UNIQUE,
  frd_document_id INTEGER REFERENCES frd_documents(id),
  request_id      INTEGER REFERENCES requests(id),
  content         JSONB NOT NULL,
  generated_by    INTEGER REFERENCES users(id),
  generated_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Important/flagged messages in chat channels
important_messages (
  id              SERIAL PRIMARY KEY,
  request_id      INTEGER REFERENCES requests(id),
  message_id      TEXT NOT NULL,          -- Stream Chat message ID
  message_text    TEXT,
  sender_id       TEXT,
  sender_name     TEXT,
  flagged_by      INTEGER REFERENCES users(id),
  flagged_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Auth event log
auth_logs (
  id              SERIAL PRIMARY KEY,
  user_id         INTEGER REFERENCES users(id),
  action          TEXT,
  ip_address      TEXT,
  user_agent      TEXT,
  timestamp       TIMESTAMPTZ DEFAULT NOW()
)

-- Admin action audit trail
admin_audit_logs (
  id              SERIAL PRIMARY KEY,
  admin_id        INTEGER,
  action          TEXT,
  details         JSONB,
  timestamp       TIMESTAMPTZ DEFAULT NOW()
)
```

Run `node backend/migrations/run.js` to initialise all tables.

## Running Locally

### Prerequisites
- Node.js 18+
- A [Neon](https://neon.tech) PostgreSQL database
- A [Supabase](https://supabase.com) project with a private storage bucket named `bprm-attachments`
- An [Azure OpenAI](https://azure.microsoft.com/en-us/products/ai-services/openai-service) resource with a `gpt-4o-mini` deployment
- A [Stream Chat](https://getstream.io) app
- A [Daily.co](https://daily.co) account (for video calls)

### Setup

```bash
# 1. Clone the repo
git clone https://github.com/Baikalsphere/nileBRD.git
cd nileBRD

# 2. Backend setup
cd backend
cp .env.example .env        # fill in all env vars
npm install
node migrations/run.js      # initialise DB schema
npm run dev                 # starts on http://localhost:5001

# 3. Frontend setup (new terminal)
cd frontend
cp .env.local.example .env.local   # set NEXT_PUBLIC_BACKEND_URL
npm install
npm run dev                         # starts on http://localhost:3000
```

### Default Accounts

Create users via the Admin Control Panel or the `/signup` page. Available roles:
- `stakeholder` — access to Stakeholder Portal
- `ba` — access to Business Analyst Portal
- `it` — access to IT Portal

Admin panel is accessed separately at `/admin-control-panel` using the `ADMIN_PASSWORD` from `.env`.

## Deployment

The app is designed for cloud deployment with the following recommended services:

| Component | Recommended Platform |
|-----------|---------------------|
| Frontend | Vercel (Next.js native) |
| Backend | Railway / Render / Fly.io |
| Database | Neon (already cloud-native) |
| File Storage | Supabase Storage |
| AI | Azure OpenAI |

### Production env changes
- Set `NODE_ENV=production`
- Set `FRONTEND_URL` to your Vercel domain
- Set `NEXT_PUBLIC_BACKEND_URL` to your backend URL
- Ensure `SUPABASE_SERVICE_KEY` is the full `service_role` JWT from Supabase project settings
