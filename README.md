# BPRM Portal — Business Process Requirements Management

A full-stack web application that automates the end-to-end business requirements lifecycle — from stakeholder problem submission through a 4-stage AI-assisted BRD generation pipeline, IT feasibility analysis, and UAT sign-off.

## Overview

The BPRM Portal connects three teams — Stakeholders, Business Analysts, and IT — on a single platform. It replaces fragmented email chains and manual Word documents with a structured, AI-powered workflow:

```
Stakeholder submits problem (with file attachments)
        ↓
BA opens Stream Chat discussion channel
        ↓
BA marks key messages → AI extracts requirements & key points
        ↓
  ┌─────────────────────────────────────────────────┐
  │           4-Stage Staged BRD Flow                │
  │  Stage 1: Completeness check (AI scores 0–100)   │
  │  Stage 2: Scope definition (BA edits & approves) │
  │  Stage 3: Workflow generation (BA edits & approves) │
  │  Stage 4: Full BRD generation (11 AI calls, parallel) │
  └─────────────────────────────────────────────────┘
        ↓
Stakeholders review BRD → approve or request changes
        ↓
IT receives approved BRD → generates FRD → generates Test Cases
        ↓
Stakeholder performs UAT → sign-off
```

All AI-generated content is **strictly grounded** in the actual discussion and uploaded documents. No hallucination, no hardcoded domain boilerplate.

## AI Features

### BRD Generation Pipeline (4-Stage Staged Flow)

The BRD pipeline is a gated, 4-stage process. The BA must complete and approve each stage before proceeding to the next. At any stage, the BA can go back to the discussion to gather more information without losing progress.

#### Stage 0: Key Point Extraction
**Endpoint:** `POST /api/stream/channels/:requestId/generate-key-points`

- BA marks important messages in the chat with a bookmark
- GPT-4o classifies all marked messages: `REQUIREMENT | CONCERN | ACTION | GENERAL`
- Extracts: key requirements, stakeholder concerns, action items, keywords, integration signals
- Produces a BRD Readiness score (0–5) and readiness checks
- Attached documents are parsed and injected into the AI context

#### Stage 1: Completeness Check
**Endpoint:** `POST /api/stream/channels/:requestId/completeness-check`

- AI scores the discussion 0–100 across 8 BRD coverage dimensions
- Returns: score, readiness verdict, what's covered, critical gaps, specific questions for the stakeholder
- BA can proceed or go back to chat to resolve gaps

#### Stage 2: Scope Definition
**Endpoint:** `POST /api/stream/channels/:requestId/generate-scope`

- AI extracts in-scope items, out-of-scope items, ambiguities, and critical gaps
- Every in-scope item is backed by a verbatim source quote from the discussion
- BA reviews, edits inline, and formally approves
- Approved scope is saved to `brd_scopes` table

#### Stage 3: Workflow Generation
**Endpoint:** `POST /api/stream/channels/:requestId/generate-workflow`

- AI generates a step-by-step end-to-end business process from the approved scope
- Each step has: actor, action, outcome, systems involved, decision points
- BA reviews, edits inline, and approves
- Approved workflow is saved to `brd_workflows` table

#### Stage 4: Full BRD Generation
**Endpoint:** `POST /api/stream/channels/:requestId/generate-brd`

Generates a comprehensive 13-section BRD document. All 11 AI calls run in **parallel** for performance. Every section is grounded strictly in the actual discussion and documents — no hardcoded domain boilerplate.

| Section | Method |
|---------|--------|
| 1. Executive Summary | AI-grounded (4–5 sentences: problem, solution, benefit, dependencies) |
| 2. Business Objective & Goals | AI-grounded SMART objective + 4–6 specific goals |
| 3. Scope | Uses BA-approved scope (Stage 2) + approved workflow process flow (Stage 3) |
| 4. Assumptions & Constraints | AI-extracted from discussion |
| 5. Stakeholder Analysis | Extracted from message participants + request metadata |
| 6. Functional Requirements | AI-grouped by area, with acceptance criteria + source quotes |
| 7. Non-Functional Requirements | AI-extracted categories with evidence from discussion |
| 8. Business Rules | AI-extracted rules, calculations, and constraints |
| 9. Integration Requirements | AI-extracted only from explicitly mentioned integrations |
| 10. Risk Register | AI-formalised risks with impact, probability, and mitigation |
| 11. Action Items | AI-formalised next steps from discussion |
| 12. BRD Readiness Assessment | Score and checks from Stage 0 |
| Appendix | Verbatim key conversation excerpts |

### BRD Enhancement
**Endpoint:** `POST /api/stream/brd-documents/:brdId/enhance`

Re-generates the BRD incorporating stakeholder review feedback. Adds new requirements from review comments, formalises new risks raised, increments version number.

### FRD Generation
**Endpoint:** `POST /api/stream/brd-documents/:brdId/generate-frd`

Expands an approved BRD into a full Functional Requirements Document. Sections: Overview, Functional Specifications (with acceptance criteria + business rules), System Behaviour, Data Requirements, UI Requirements, Integration Requirements, NFRs, Traceability Matrix.

### Test Case Generation
**Endpoint:** `POST /api/stream/frd-documents/:frdId/generate-test-cases`

Generates comprehensive test suites from FRD functional specifications. Test types: Critical, System, UAT, Regression. Each case includes steps, expected outcomes, priority, and type.

### PDF Export

Both BRD and FRD/TC exports use a professional `window.print()` design:
- Dark gradient cover pages with document metadata grids
- Dark-header tables, priority-bordered requirement cards
- Two-column acceptance criteria layout
- Auto table-of-contents generation
- Print-colour-adjust for accurate colour reproduction

## Database Schema

```sql
-- Core users
users (
  id              SERIAL PRIMARY KEY,
  email           TEXT UNIQUE NOT NULL,
  password_hash   TEXT NOT NULL,
  name            TEXT,
  role            TEXT NOT NULL,          -- 'stakeholder' | 'ba' | 'it'
  is_it_manager   BOOLEAN DEFAULT false,
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Business requests submitted by stakeholders
requests (
  id              SERIAL PRIMARY KEY,
  req_number      TEXT UNIQUE,            -- e.g. REQ-0001
  title           TEXT NOT NULL,
  description     TEXT,
  category        TEXT,
  priority        TEXT,
  status          TEXT DEFAULT 'Open',
  stakeholder_id  INTEGER REFERENCES users(id),
  assigned_to     INTEGER REFERENCES users(id),
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- File attachments on requests (stored in Supabase)
request_attachments (
  id              SERIAL PRIMARY KEY,
  request_id      INTEGER REFERENCES requests(id),
  original_name   TEXT,
  mimetype        TEXT,
  size            INTEGER,
  s3_key          TEXT,
  uploaded_by     INTEGER REFERENCES users(id),
  created_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Important messages bookmarked by BA for BRD analysis
important_messages (
  id                  SERIAL PRIMARY KEY,
  request_id          INTEGER REFERENCES requests(id),
  stream_message_id   TEXT NOT NULL,
  message_text        TEXT,
  sender_name         TEXT,
  marked_by           INTEGER REFERENCES users(id),
  marked_at           TIMESTAMPTZ DEFAULT NOW()
)

-- BA-approved scope definitions (Stage 2)
brd_scopes (
  id              SERIAL PRIMARY KEY,
  request_id      INTEGER REFERENCES requests(id),
  content         JSONB NOT NULL,         -- {scope_title, in_scope, out_of_scope, ambiguities, critical_gaps}
  status          TEXT DEFAULT 'draft',   -- 'draft' | 'approved'
  created_by      INTEGER REFERENCES users(id),
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  approved_at     TIMESTAMPTZ
)

-- BA-approved process workflows (Stage 3)
brd_workflows (
  id              SERIAL PRIMARY KEY,
  request_id      INTEGER REFERENCES requests(id),
  scope_id        INTEGER REFERENCES brd_scopes(id),
  content         JSONB NOT NULL,         -- {workflow_title, steps[]}
  status          TEXT DEFAULT 'draft',   -- 'draft' | 'approved'
  created_by      INTEGER REFERENCES users(id),
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  approved_at     TIMESTAMPTZ
)

-- BRD documents (Stage 4 output)
brd_documents (
  id              SERIAL PRIMARY KEY,
  doc_id          TEXT UNIQUE,            -- e.g. BRD-REQ0001-v0.1
  request_id      INTEGER REFERENCES requests(id),
  version         TEXT DEFAULT '0.1',
  status          TEXT DEFAULT 'Draft',   -- Draft | In Review | Approved | Final
  content         JSONB NOT NULL,         -- full 13-section BRD JSON
  generated_by    INTEGER REFERENCES users(id),
  generated_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Stakeholder reviews on BRDs
brd_reviews (
  id                  SERIAL PRIMARY KEY,
  brd_document_id     INTEGER REFERENCES brd_documents(id),
  reviewer_id         INTEGER REFERENCES users(id),
  reviewer_name       TEXT,
  status              TEXT,               -- 'pending' | 'approved' | 'changes_requested'
  comment             TEXT,
  reviewed_at         TIMESTAMPTZ DEFAULT NOW()
)

-- Tracks BRDs routed to IT manager
brd_it_submissions (
  id                  SERIAL PRIMARY KEY,
  brd_document_id     INTEGER REFERENCES brd_documents(id),
  submitted_by        INTEGER REFERENCES users(id),
  it_manager_id       INTEGER REFERENCES users(id),
  submitted_at        TIMESTAMPTZ DEFAULT NOW()
)

-- FRD documents (generated by IT)
frd_documents (
  id              SERIAL PRIMARY KEY,
  doc_id          TEXT UNIQUE,
  brd_document_id INTEGER REFERENCES brd_documents(id),
  request_id      INTEGER REFERENCES requests(id),
  version         TEXT DEFAULT '1.0',
  status          TEXT DEFAULT 'Draft',
  content         JSONB NOT NULL,
  generated_by    INTEGER REFERENCES users(id),
  generated_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Test case suites
test_case_documents (
  id              SERIAL PRIMARY KEY,
  doc_id          TEXT UNIQUE,
  frd_document_id INTEGER REFERENCES frd_documents(id),
  request_id      INTEGER REFERENCES requests(id),
  content         JSONB NOT NULL,
  generated_by    INTEGER REFERENCES users(id),
  generated_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
)

-- Admin action audit trail
admin_audit_logs (
  id              SERIAL PRIMARY KEY,
  admin_id        INTEGER,
  action          TEXT,
  details         JSONB,
  timestamp       TIMESTAMPTZ DEFAULT NOW()
)
```

Run `node backend/migrations/run.js` to initialise all tables.

## Running Locally

### Prerequisites
- Node.js 18+
- [Neon](https://neon.tech) PostgreSQL database
- [Supabase](https://supabase.com) project with a private storage bucket named `bprm-attachments`
- [Azure OpenAI](https://azure.microsoft.com/en-us/products/ai-services/openai-service) resource with a `gpt-4o` deployment
- [Stream Chat](https://getstream.io) app
- [Daily.co](https://daily.co) account (for video calls)

### Setup

```bash
# 1. Clone the repo
git clone https://github.com/Baikalsphere/nileBRD.git
cd nileBRD

# 2. Backend setup
cd backend
cp .env.example .env        # fill in all env vars
npm install
node migrations/run.js      # initialise DB schema
npm run dev                 # starts on http://localhost:5001

# 3. Frontend setup (new terminal)
cd frontend
cp .env.local.example .env.local
npm install
npm run dev                 # starts on http://localhost:3000
```

### Default Accounts

Create users via the Admin Control Panel (`/admin-control-panel`) or the `/signup` page. Roles:
- `stakeholder` — Stakeholder Portal
- `ba` — Business Analyst Portal
- `it` — IT Portal
